🧬 Прямая валидация
в базах данных: ОСС, MSSQL, целостность

От распределённых реестров до байтового парсинга — как СУБД гарантирует истину

Если рассматривать ОСС с прямой валидацией через призму баз данных, процесс превращается в сложную систему синхронизации и верификации между государственными и частными реестрами. Это переход от «декларативного» хранения к «верифицируемому», где каждая запись подтверждается динамической связью с мастер-данными.

🏛️ Федеративная архитектура

ЕСИА (identity), Росреестр (ЕГРН) (права), ГИС ЖКХ — runtime-запросы через API, а не единое хранилище.

🔗 Ссылочная целостность

Голос в БД собрания не может существовать без валидной ссылки на запись в ЕГРН. Каждая транзакция проверяет актуальность собственника.

🧾 Неизменность (Immutability)

Append-only логи, хеширование голосов. Любое изменение — новая транзакция с цифровой подписью; итоговый хеш защищает протокол.

⚙️ Автоматизация кворума

Триггеры и хранимые процедуры пересчитывают сумму долей, а JOIN с реестром всегда возвращает актуальные метры.

📌 Глубинная суть: инкапсуляция, атомарность, независимость

Прямая валидация — это философия недоверия. База данных не верит приложению и проверяет всё сама. Три фундамента:

  • Инкапсуляция бизнес-логики в схему — ограничения CHECK, типы данных, FOREIGN KEY вшивают правила прямо в структуру.
  • Единая точка истины (Single Source of Truth) — валидация на уровне БД защищает данные от любого клиента (web, mobile, скрипты).
  • Защита от состояния гонки — UNIQUE constraint работает внутри ядра под замком, предотвращая дубликаты даже при параллельных запросах.
💡 «Декларативный подход (CHECK price > 0) против императивного: вы описываете, каким должен быть результат, а не как проверять. Это быстрее и надёжнее, потому что оптимизировано на уровне движка СУБД.»

⚡ Microsoft SQL Server — многослойный «бункер» целостности

В MSSQL прямая валидация выходит на новый уровень благодаря интеграции с .NET, строгой типизации и расширенным ограничениям.

CHECK + пользовательские функции (UDF) SQL CLR (валидация на C#) Типы hierarchyid, geography INSTEAD OF триггеры XML Schema Collection Строгая DRI (FOREIGN KEY)

🗺️ Пример: тип geography и байтовый парсинг

Тип geography хранит координаты. Сервер берёт входящую строку, читает её побайтово: первый байт — тип фигуры, затем широта/долгота. Если широта > 90° или структура бинарного формата нарушена — парсинг прерывается, и данные даже не попадают на диск. Это прямая валидация на уровне байтов.

Аналогично работают hierarchyid, datetimeoffset (проверка смещения часового пояса) и XML со схемой — MSSQL проверяет каждый элемент на соответствие XSD.

🔐 Прямая валидация на уровне транзакций и страниц

Валидация встроена в жизненный цикл транзакции — от предварительной проверки до физической записи на диск.

📋 Предварительная валидация (In-Flight)

При вставке 1000 строк, если 500-я нарушает ограничение, транзакция прерывается немедленно. Экономия ресурсов.

🔄 Теневые страницы (Versioning)

Snapshot Isolation: при COMMIT проверяется, не изменились ли прочитанные данные. Если да — откат.

📀 Transaction Hardening

LSN и подтверждение от контроллера диска. Транзакция не считается завершённой, пока байты физически не записаны.

🧮 Контрольная сумма страниц

При чтении страницы (8 КБ) SQL Server пересчитывает контрольную сумму и сравнивает с заголовком. Ошибка 824 при несовпадении — защита от битых секторов.

🧱 Алгебраизатор, оптимизатор и байтовый парсинг

На максимальной глубине валидация — это многоуровневый конвейер:

  • Algebrizer & Binding — связывает имена объектов с метаданными, проверяет типы ещё до выполнения.
  • Оптимизатор строит план, вшивая проверки CHECK и FK как предикаты.
  • Latch (защелка) гарантирует, что во время проверки байты не изменятся другим потоком.
  • Байтовый парсинг для типов вроде datetimeoffset — смещение не может быть +14:01, иначе ошибка.
  • Constraint Engine выполняет логические проверки.
  • WAL (Write-Ahead Log) — запись в лог до изменения страницы, гарантия отката.

🛡️ Контрольные суммы и защита от повреждения железа

Когда SQL Server записывает страницу (8 КБ), он вычисляет контрольную сумму всех байтов и сохраняет её в заголовке. При чтении сумма пересчитывается — если хотя бы один бит «поплыл», сервер выдаёт ошибку 824. Это прямая валидация целостности на физическом уровне.

Включение опции PAGE_VERIFY CHECKSUM (по умолчанию) — последний рубеж обороны данных.

🗳️ ОСС с прямой валидацией — схема данных

Как упоминалось в документе, архитектура подразумевает динамическую связь таблиц голосования с мастер-данными ЕГРН и ГИС ЖКХ. Ниже концептуальная ER-связь (упрощённо):

┌─────────────────┐     ┌──────────────────────┐     ┌─────────────────┐
│   ЕСИА (users)  │     │   Голосование (votes) │     │  ЕГРН (premises)│
├─────────────────┤     ├──────────────────────┤     ├─────────────────┤
│ user_id (PK)    │◄────│ user_id (FK)          │     │ cad_num (PK)    │
│ full_name       │     │ cad_num (FK)          │────►│ owner_id        │
│ snils / inn     │     │ vote_value            │     │ area_share      │
└─────────────────┘     │ vote_hash             │     │ address         │
                        │ timestamp (append-only)│     └─────────────────┘
                        └──────────────────────┘

Ссылочная целостность: голос существует только при подтверждённом праве собственности из ЕГРН.

🔎 Философия недоверия — итог

Прямая валидация в БД (особенно в MSSQL) — это не просто «галочка». Это многослойная защита: от синтаксиса до контрольной суммы на диске. Данные становятся «умными» — они сами знают, какими могут быть. ОСС с прямой валидацией превращает голосование в верифицируемый, защищённый от подделок процесс, опирающийся на государственные мастер-данные.
